## 四、通信协议

所有消息均为 UTF-8 编码的 JSON 字符串，通过 WebSocket 传输。

### 4.1 消息类型常量

```
被控端 → 主控端：
  register        注册（连接后立即发送，携带主机名/IP/MAC）
  heartbeat       心跳（每 20 秒，携带过滤状态和网络状态）
  status          状态上报（响应主控端的 get_status 请求）
  ack             操作确认
  browsing_update 浏览域名上报（每 15 秒，携带最近 DNS 查询列表）

主控端 → 被控端：
  update_rules    下发规则（域名列表、模式、LAN 网段、DNS、密码 hash）
  set_filter      启用/禁用过滤（携带模式 whitelist/blacklist）
  disconnect      断网指令
  reconnect       恢复上网指令
  get_status      请求状态上报
```

### 4.2 关键消息字段

**update_rules（下发规则）**
```json
{
  "type": "update_rules",
  "domains":          ["baidu.com", "*.baidu.com"],
  "lan_subnets":      ["192.168.1.0/24"],
  "controller_ip":    "192.168.1.100",
  "upstream_dns":     "114.114.114.114",
  "mode":             "whitelist",
  "tray_pwd_hash":    "<sha256>",
  "unlock_pwd_hash":  "<sha256>"
}
```

**set_filter**
```json
{ "type": "set_filter", "enabled": true, "mode": "whitelist" }
```

**browsing_update**
```json
{
  "type": "browsing_update",
  "domains": [
    { "domain": "www.baidu.com", "ts": "10:23:45" },
    ...
  ]
}
```

### 4.3 网络状态值

```
normal      正常上网
whitelist   白名单过滤中
blacklist   黑名单过滤中
disconnect  已断网
offline     已断开（仅主控端侧，表示与主控端连接已断开）
```

## 五、双层过滤机制

### 5.1 DNS 层（所有模式）

`FilterResolver.resolve()` 在每次 DNS 查询时：

```
收到查询 qname
│
├─ 调用 on_query(domain) 回调 → 记录浏览历史
│
├─ _matches(qname)：遍历规则列表
│    *.example.com → name == "example.com" 或 name.endswith(".example.com")
│    example.com   → name == "example.com" 或 name.endswith(".example.com")
│
白名单模式：
  matched=True  → _forward(upstream_dns)  正常解析
  matched=False → NXDOMAIN                域名"不存在"

黑名单模式：
  matched=True  → NXDOMAIN
  matched=False → _forward(upstream_dns)
```

DNS 服务器绑定 `127.0.0.1:53`，通过 PowerShell `Set-DnsClientServerAddress` 将网卡 DNS 强制改为 `127.0.0.1`，所有 DNS 查询均经过本地过滤服务器。

### 5.2 路由表层（白名单 + 断网专用）

**为什么放弃防火墙方案？**

Windows 防火墙存在预装的 Chrome / Edge 出站 Allow 规则，优先级高于用户添加的 Block 规则，导致防火墙默认封锁策略被绕过。

**当前方案：删除默认网关路由**

```
断网状态：
  route DELETE 0.0.0.0 MASK 0.0.0.0
  → 路由表中无默认路由 → 所有非局域网流量无路可走
  → LAN 子网路由（192.168.x.x/24）天然保留，内网通信不受影响

白名单状态：
  同样删除默认路由（断网效果）
  + 为白名单 IP / 上游 DNS / 主控端 各添加 /32 主机路由
  → 只有明确放行的 IP 有路由
  → 绑过 DNS 直连 IP 同样无效（那个 IP 不在主机路由表中）
```

### 5.3 白名单路由配置的三个阶段

`apply_whitelist_routing()` 的核心挑战：解析域名时需要网络，但白名单模式要求断网。通过三阶段解决：

```
Phase 1 — 确保有默认路由
  _get_default_gateways() 返回空（已断网状态）
  → 调用 reconnect_internet()，从 gw_backup.json 恢复默认路由
  → 验证路由表已有默认路由

Phase 2 — 网卡 DNS 临时切换到上游，直接解析
  set_adapter_dns(upstream_dns)   ← 绕过本地 DNS 服务器
  Clear-DnsClientCache
  socket.getaddrinfo(domain)      ← Windows DNS Client 直查上游
  → 获得白名单域名的真实 IP 列表

Phase 3 — 切回本地 DNS，加主机路由，再断网
  set_adapter_dns("127.0.0.1")
  _add_host_routes_bulk(ips)      ← 一次 PowerShell 调用批量添加（避免耗时过长）
  disconnect_internet()           ← 路由就绪后才删默认路由
```

关键顺序：**先加完所有主机路由，再删默认路由**。若顺序相反，删默认路由后的几十秒内上游 DNS 主机路由尚未加好，本地 DNS 服务器转发查询会报 WinError 10051（网络不可达）。

### 5.4 路由备份与恢复

```
disconnect_internet()：
  1. 读当前 Get-NetRoute 0.0.0.0/0 → 保存到 gw_backup.json
  2. route DELETE 0.0.0.0 MASK 0.0.0.0

reconnect_internet()：
  1. 读 gw_backup.json → New-NetRoute 还原每条路由
     （RouteMetric=0 时省略 -RouteMetric 参数，让 Windows 自动选择）
  2. 路由添加后验证 Get-NetRoute 确认存在
  3. 成功后删除 gw_backup.json
```

## 六、被控端状态机

`AgentCore._set_state(new_state)` 是状态切换的唯一入口：

```
STATE_NORMAL
  _do_restore()
    → 停 DNS 服务器
    → 恢复网卡 DNS（DHCP 自动获取）
    → 清除 NC_ 防火墙规则，设默认出站 Allow
    → reconnect_internet()（恢复默认网关路由）

STATE_WHITELIST
  _do_enable_whitelist()
    → 停旧 DNS 服务器
    → 启动新 DNS 服务器（whitelist 模式）
    → set_adapter_dns("127.0.0.1")
    → apply_whitelist_routing()（三阶段路由配置）

STATE_BLACKLIST
  _do_enable_blacklist()
    → 停旧 DNS 服务器
    → reconnect_internet()（清理白名单/断网模式可能遗留的主机路由）
    → 清除 NC_ 防火墙规则
    → 启动新 DNS 服务器（blacklist 模式）
    → set_adapter_dns("127.0.0.1")

STATE_DISCONNECT
  _do_disconnect()
    → 停 DNS 服务器
    → 恢复网卡 DNS
    → 清除 NC_ 防火墙规则
    → disconnect_internet()（删默认路由，主机路由若有则一并清理）
```

热更新（过滤中收到新规则）：
- 当前 whitelist + 新规则 whitelist → `dns_server.update_domains()` + `apply_whitelist_routing()`
- 当前 blacklist + 新规则 blacklist → 仅 `dns_server.update_domains()`

## 七、主控端被控端状态管理

### 7.1 在线连接池与离线记录池

`ControllerServer` 维护两个字典：

```python
_agents: dict[str, ConnectedAgent]   # 当前已连接
_offline: dict[str, ConnectedAgent]  # 曾连接但已断开
```

**连接时**：检查 `_offline` 中是否有该 IP 的历史记录；若有，复用（hostname、mac、recent_domains 等保留），将其移回 `_agents`。

**断开时**：从 `_agents` 移出，标记 `is_online=False`，放入 `_offline`。

`get_agents()` 返回在线 + 离线全部，在线在前。GUI 用 `is_online` 字段区分显示。

### 7.2 ConnectedAgent 字段

| 字段 | 说明 |
|---|---|
| `ip` | 被控端 IP |
| `hostname` | 主机名（注册时上报） |
| `mac` | MAC 地址 |
| `is_online` | 当前是否在线 |
| `net_state` | 网络状态（normal/whitelist/blacklist/disconnect） |
| `filter_active` | 是否处于过滤模式 |
| `connected_at` | 最近一次连接时间 |
| `last_seen` | 最近一次心跳时间 |
| `recent_domains` | 最近访问域名（deque maxlen=30） |

**列定义**（COL_CHECK ~ COL_BROWSE）：

### 8.3 线程模型

```
Qt 主线程
  └── PyQt6 事件循环（GUI 渲染、信号/槽）
```

## 九、浏览监控

DNS 解析的每次查询触发 `on_query` 回调，过滤规则如下：

```python
if "." in clean and not clean.endswith(".local") and not clean.startswith("_"):
    on_query(clean)
```

过滤掉 mDNS（`.local`）和 SRV 记录（`_xxx`），只保留真实域名访问。

查询记录存入 `AgentCore._recent_domains`（`collections.deque(maxlen=50)`），由 `threading.Lock` 保护线程安全。

`AgentWSClient._browsing_loop` 每 15 秒将记录批量发送给主控端，主控端 `ConnectedAgent.recent_domains`（`deque(maxlen=30)`）保存最新状态并通过 `to_dict()` 提供给 GUI。

## 十、网线断开检测与锁屏

### 10.1 检测逻辑

`AgentCore._monitor_network()` 是一个 asyncio 协程，启动后每 3 秒轮询一次：

```python
result = subprocess.run(
    ['powershell', '-Command',
     '(Get-NetAdapter | Where-Object {$_.Status -eq "Up" -and '
     '$_.Name -notmatch "Loopback"}).Count'],
    ...
)
```

返回 0 → 所有网卡断开 → 触发锁屏；大于 0 → 恢复正常。

### 10.2 Session 0 隔离问题

Windows 服务运行在 Session 0（无桌面会话），直接调用 `subprocess.Popen` 创建 GUI 进程时窗口不会显示在用户桌面。

解决方案：使用 `win32ts.WTSQueryUserToken` 获取当前活跃用户 Session 的 Token，`win32security.DuplicateTokenEx` 创建主令牌，`win32process.CreateProcessAsUser` 在用户 Session（`winsta0\default`）中创建进程：

```python
session_id    = win32ts.WTSGetActiveConsoleSessionId()
user_token    = win32ts.WTSQueryUserToken(session_id)
primary_token = win32security.DuplicateTokenEx(
    user_token, TOKEN_ALL_ACCESS, None,
    SecurityImpersonation, TokenPrimary
)
si = win32process.STARTUPINFO()
si.lpDesktop = "winsta0\\default"
win32process.CreateProcessAsUser(
    primary_token, None, f'"{exe}" --lock "{hash}"',
    None, None, False, NORMAL_PRIORITY_CLASS, None, None, si
)
```

调试模式下（非 SYSTEM 权限，`WTSQueryUserToken` 报错 1314）自动降级为 `subprocess.Popen` 在当前会话启动。

### 10.3 锁屏界面实现

`LockScreen`（PyQt6）：
- `FramelessWindowHint | WindowStaysOnTopHint` 标志使窗口全屏无边框且始终置顶
- QTimer 每秒触发 `_tick()`，检查网络状态
- 密码验证：`hashlib.sha256(input.encode()).hexdigest() == password_hash`
- 超时关机：`subprocess.run(["shutdown", "/s", "/t", "0"])`
- 拦截 Esc 和 Alt+F4：重写 `keyPressEvent`

## 十一、数据库设计

主控端使用 SQLite（`controller.db`），所有操作通过 `threading.Lock` 串行化。

```sql
CREATE TABLE whitelist (
    id      INTEGER PRIMARY KEY AUTOINCREMENT,
    name    TEXT NOT NULL,
    domain  TEXT NOT NULL UNIQUE,
    enabled INTEGER NOT NULL DEFAULT 1

CREATE TABLE blacklist (
    id      INTEGER PRIMARY KEY AUTOINCREMENT,
    name    TEXT NOT NULL,
    domain  TEXT NOT NULL UNIQUE,
    enabled INTEGER NOT NULL DEFAULT 1

CREATE TABLE settings (
    key   TEXT PRIMARY KEY,
    value TEXT
);
-- 默认 settings：controller_port, upstream_dns, lan_subnets,

```

密码均以 SHA256 hex 字符串存储，明文不落库。

## 十二、打包说明

使用 PyInstaller `--onefile` 模式，打包配置分别在 `controller.spec` 和 `agent.spec`。

**关键注意事项：**

1. **`win32timezone`** 必须在 `hiddenimports` 中显式声明，否则 pywin32 服务注册会在运行时报"找不到时区"错误。

2. **`agent.ui.lock_screen`** 在 `windows_service.py` 中通过条件 import 引用，PyInstaller 静态分析不到，必须手动列入。

3. **`win32ts` / `win32process`** 同上，用于 Session 隔离启动，必须显式声明。

4. **路径适配**：`shared/paths.py` 的 `get_app_dir()` 通过 `sys.frozen` 区分打包模式（返回 `os.path.dirname(sys.executable)`）和开发模式（返回项目根目录），确保 `config.json` 和 `controller.db` 始终与 EXE 同目录。

5. **`collect_all()`**：websockets、pystray、dnslib 使用 `collect_all()` 收集所有子模块和数据文件，避免运行时 import 失败。

6. **被控端使用 `console=True`**：服务模式下控制台窗口不显示（SCM 启动），但保留控制台便于 `--lock` 模式调试；主控端使用 `console=False` 避免出现黑色命令窗口。

**打包命令：**
```
pyinstaller controller.spec --clean --noconfirm
pyinstaller agent.spec      --clean --noconfirm
copy agent\config.json dist\config.json
```

## 十三、远程密码同步机制

`update_rules` 消息携带两个可选字段 `tray_pwd_hash` 和 `unlock_pwd_hash`（空字符串表示不更新）。主控端每次推送规则时从数据库读取当前 hash 并附带发送。

被控端收到后：
1. 非空 → 覆盖 `self.config` 中对应字段
2. 托盘密码立即热更新（`self.tray.password_hash = tray_pwd_hash`）
3. 锁屏密码在下次拔网线触发时生效（`_launch_lock_screen` 从 `self.config` 读取）
4. 调用 `save_config()` 写入 `config.json`，服务重启后仍然有效

## 十四、已知限制

| 限制 | 原因 | 影响 |
|---|---|---|
| 白名单 IP 放行为静态快照 | 启用时解析一次域名取 IP | CDN IP 会变，长时间后可能需重新推送规则以更新路由 |
| 黑名单无路由层 | 黑名单仅拦截 DNS，不修改路由表 | 学生直接用 IP 访问可绕过黑名单（白名单不受影响） |
| 浏览记录仅在过滤模式下采集 | DNS 查询只经过本地 DNS 时才有回调 | 正常模式下无法监控浏览行为 |
| 锁屏依赖活跃用户 Session | `WTSGetActiveConsoleSessionId` | 无用户登录时（纯服务器环境）锁屏无法显示 |
| 白名单启用时短暂可完整上网 | Phase 1~2 期间恢复默认路由以便解析域名 | 持续约 2~5 秒，教室场景下学生来不及反应 |
